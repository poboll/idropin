### 5. 常见问题准备

#### 技术问题

**Q1: 为什么选择PostgreSQL而不是MySQL？**

A:
1. PostgreSQL 16的JSONB类型更适合存储文件元数据
2. 原生数组类型（TEXT[]）比MySQL的JSON更高效
3. 强大的全文搜索能力（tsvector + GIN索引）
4. 更好的并发控制和MVCC机制
5. 丰富的扩展生态和高级特性

**Q2: 为什么选择Spring Boot而不是Node.js？**

A:
1. Spring生态系统成熟稳定，适合企业级应用
2. 强大的类型系统和编译时错误检查
3. 丰富的开发工具和IDE支持（IntelliJ IDEA）
4. 大厂技术栈更符合求职要求
5. 更好的性能和可扩展性

**Q3: 为什么选择Next.js而不是React？**

A:
1. Next.js 14提供SSR和更好的SEO
2. 内置路由和代码分割，无需额外配置
3. 更好的开发体验和热更新
4. 服务端渲染支持，首屏加载更快
5. 丰富的生态和社区支持

**Q4: 如何实现大文件分片上传？**

A:
1. 前端将大文件分割为5MB的分片
2. 使用MD5计算文件哈希值
3. 依次上传每个分片到服务器
4. 服务器端合并分片并验证MD5
5. 支持断点续传，网络中断后可继续上传
6. 支持秒传，相同文件直接秒传

**Q5: 如何实现PWA离线功能？**

A:
1. 使用Service Worker缓存静态资源
2. 使用IndexedDB存储离线数据
3. 监听网络状态，在线时同步数据
4. 离线上传的文件存储在队列中
5. 网络恢复后自动同步队列
6. 提供离线访问已缓存的文件

**Q6: 如何实现全文搜索？**

A:
1. PostgreSQL 16的全文搜索（tsvector）
2. 创建GIN索引加速查询
3. 支持关键词搜索（name, originalName）
4. 支持MIME类型过滤
5. 支持分类和标签过滤
6. 支持文件大小和日期范围过滤
7. 提供搜索建议功能

**Q7: 如何实现实时数据分析？**

A:
1. 使用聚合查询统计文件数量和存储空间
2. 使用日期范围查询统计上传趋势
3. 使用GROUP BY统计文件类型分布
4. 使用JOIN查询统计分类文件数
5. 使用Redis缓存热点统计数据（5分钟TTL）
6. 前端使用Recharts绘制可视化图表

**Q8: 测试覆盖率如何达到85%？**

A:
1. 编写64个测试用例（24个单元测试 + 40个集成测试）
2. 使用JUnit 5和Mockito进行单元测试
3. 使用MockMvc进行Controller层测试
4. 使用Spring Boot Test进行集成测试
5. 使用JaCoCo生成测试覆盖率报告
6. 覆盖Service层、Controller层和工具类

#### 遇到的挑战和解决方案（3-5分钟）

**挑战1：大文件分片上传的断点续传实现**

**问题**：
- 网络中断后如何继续上传
- 如何验证分片完整性
- 如何实现秒传功能

**解决方案**：
- 前端计算文件MD5，上传前检查是否已存在
- 后端验证MD5，存在则直接返回成功
- 使用分片序号记录上传进度
- 网络中断后可从断点继续上传

**挑战2：PWA离线功能的Service Worker配置**

**问题**：
- Service Worker生命周期管理
- IndexedDB数据存储和查询
- 网络状态监听和自动同步
- 缓存策略和失效机制

**解决方案**：
- 使用Workbox库简化Service Worker开发
- 实现离线队列和同步机制
- 添加缓存版本控制
- 提供手动同步选项

**挑战3：PostgreSQL 16 JSONB和数组类型的映射**

**问题**：
- JSONB类型在MyBatis Plus中的映射
- TEXT[]数组类型的TypeHandler
- 复杂查询的性能优化

**解决方案**：
- 创建StringArrayTypeHandler处理TEXT[]数组
- 使用@TableName注解指定表名
- 优化查询语句，使用原生JSON操作符
- 创建GIN索引加速JSONB和数组查询

## 答辩技巧

### 时间管理

1. **严格控制时间**
   - 开场白不超过3分钟
   - 项目概述不超过5分钟
   - 系统演示不超过30分钟
   - 技术讲解不超过15分钟
   - 成果展示不超过8分钟
   - 挑战和总结不超过5分钟
   - 总时间控制在20-25分钟

2. **重点突出**
   - 前3分钟讲最重要的内容
   - 后3分钟总结最重要的成果
   - 确保核心创新点被充分展示

3. **准备备用方案**
   - 演示环境可能出问题的准备
   - 准备备用演示视频或截图
   - 准备关键代码片段的纸质版

### 回答问题技巧

1. **听清问题再回答**
   - 不要急于回答
   - 确保理解问题的核心
   - 可以请求老师重复或澄清

2. **结构化回答**
   - 使用"首先、其次、最后"的逻辑
   - 分点回答，条理清晰
   - 每个观点不超过3句话

3. **诚实回答**
   - 不知道的就说不知道
   - 不要强行解释
   - 可以说明这是需要进一步研究的地方

4. **展示技术深度**
   - 回答技术问题时展示代码示例
   - 说明技术选型的理由
   - 展示对技术的深入理解

5. **联系项目实际**
   - 用项目中的实际例子说明
   - 引用具体的功能实现
   - 展示遇到的实际问题和解决方案

### 演示技巧

1. **提前测试**
   - 答辩前多次测试演示流程
   - 确保所有功能正常工作
   - 准备测试数据
   - 测试网络连接稳定性

2. **准备环境**
   - 清理浏览器缓存和Cookie
   - 关闭不必要的应用
   - 准备好演示账号和数据
   - 确保屏幕分辨率适合投影

3. **演示节奏**
   - 每个功能演示时间适中
   - 边演示边讲解
   - 重点功能详细演示
   - 次要功能快速演示

4. **备用方案**
   - 准备截图作为备用
   - 准备关键代码的纸质版
   - 准备演示视频（如果允许）

5. **应对突发情况**
   - 演示失败时保持冷静
   - 快速定位问题并尝试解决
   - 如果无法解决，跳过该功能继续下一个
   - 不要慌张，保持专业态度

### PPT制作技巧

1. **简洁专业**
   - 每页不超过6个要点
   - 使用简洁的语言，避免长篇大论
   - 使用图表和流程图代替大量文字
   - 统一配色和字体

2. **突出重点**
   - 创新点使用不同颜色或图标标注
   - 技术难点使用强调色或加粗
   - 成果数据使用图表展示

3. **代码展示**
   - 选择核心代码片段（不超过10个）
   - 使用语法高亮
   - 添加必要的注释
   - 控制字体大小，确保可读

4. **图表使用**
   - 使用架构图展示系统设计
   - 使用流程图展示关键流程
   - 使用ER图展示数据库设计
   - 使用统计图表展示数据分析成果
